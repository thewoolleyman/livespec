The shape of the gap, more precisely

The asymmetry you've named is genuine and structural, not just a missing skill. The two directions of drift have categorically different detectability:

Spec → impl drift. The spec is prescriptive — it enumerates rules, constraints, contracts. For each rule, you can mechanically check "is there evidence of this rule in the impl?" That's what capture-spec-gaps does. The rule-set is finite and walkable.

Impl → spec drift. The impl is descriptive — every line of code is something the impl does. You can't iterate "for each impl behavior, is it in spec?" because there's no complete enumeration of "impl behaviors" to begin with. The signal-to-noise ratio is brutal — almost everything in the impl is "not in spec" by some interpretation; almost none of it is load-bearing drift.

This means we shouldn't pretend the two directions need symmetric skills. They're not symmetric problems.

The existing doctor LLM-phase check that loses its home

Worth surfacing: today's livespec:doctor LLM-driven subjective phase already has "Spec↔implementation drift" as one of its baked checks (per the doctor SKILL.md, Step 9). So this isn't a new requirement we're inventing — it's an existing check that the new architecture orphans, because the new architecture's hard wall says livespec-core's doctor doesn't know about implementations.

So we have a concrete artifact to relocate, not just an abstract hole.

What the memo path actually covers

The memo workflow we just designed handles a substantial chunk of this:

  • During impl work, an agent or human notices "impl does X, spec doesn't mention X; spec might need to say something."
  • capture-memo "<observation>".
  • Later, process-memos triages it as spec-bound — routes via /livespec:propose-changes.
  • Becomes a proposed-change, then revise, then spec is updated.

What memos don't cover: silent drift

The memo path requires somebody noticing. What memos can't catch is silent drift — cases where:

  • The impl evolves in ways that nobody flags
  • The spec slowly becomes inaccurate over time without anyone naming a specific divergence
  • A piece of impl is correct-by-convention but the convention was never codified into spec
  • A bugfix landed that revealed spec-wrongness, but the bugfix-author didn't notice/note it

This is the residue. It's structurally hard to catch via memos because it requires somebody to notice the spec is wrong, which requires knowing what the spec currently says, which requires loading and rereading the spec — exactly the friction the memo workflow tries to avoid.

Three options for closing the residue

Option A: Trust the memo path; accept silent-drift risk

Don't add infrastructure. The memo path covers the noticed cases; for the silent cases, accept that spec slowly atrophies and rely on periodic full-spec review (manual). Pragmatic but limited — spec quality degrades over time without active intervention.

  • Promoted findings → /livespec:propose-changes (cross-namespace handoff, identical pattern to process-memos's spec-bound disposition).
