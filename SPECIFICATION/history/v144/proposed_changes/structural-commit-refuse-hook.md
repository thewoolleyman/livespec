## Proposal: Structural commit-refuse hook

### Target specification files

- SPECIFICATION/non-functional-requirements.md
- SPECIFICATION/contracts.md

### Summary

Rewrite the commit-refuse hook contract for the STRUCTURAL mechanism (refuse when `git rev-parse --git-dir` equals `git rev-parse --git-common-dir`; a secondary worktree's git-dir differs) with a `livespec.sandboxExempt` git-config Exemption marker the hook BODY reads, retiring the `livespec.primaryPath` arming step. Three edits land in `non-functional-requirements.md`: (1) the Primary-checkout commit-refuse hook rule under §"Workflow discipline — spec-side changes" — replace the `show-toplevel`/`primaryPath` mechanism description with the structural detection, ARMED ON INSTALL with no fail-open window, plus a new paragraph stating the `livespec.sandboxExempt` Exemption the body reads; (2) §"Commit-refuse hook bootstrap procedure" — drop the `livespec.primaryPath` set from the contract clauses and replace the illustrative canonical-body snippet with the structural-plus-sandboxExempt body that delegates to lefthook; (3) §"Conformance Pattern" concern #1 (Worktree-discipline) — reword the Verifier/Exemption to the confirmed Option-A design (one uniform body installed everywhere including the sandbox; the Verifier stays simple and accepts both the structural and the legacy body through the fleet migration; the Exemption is the `livespec.sandboxExempt` marker the hook BODY reads). One edit in `contracts.md`: §"master-direct-uncommitted-spec-edits" — repair the parenthetical that explained the OLD `show-toplevel` mechanism so it describes the structural git-dir==git-common-dir refuse (internal consistency; the invariant's conclusion is unchanged).

### Motivation

Increment 5, milestone M2 of epic livespec-zs22 (the Conformance Pattern; sub-epic livespec-zs22.7, milestone livespec-zs22.7.3 — the M2-2 core PR). The Worktree-discipline concern's Mechanism is the structural, armed-on-install commit-refuse body that supersedes the `livespec.primaryPath` arming step — the console-incident root cause was an installed-but-unarmed hook that read as protected but silently no-op'd, failing OPEN. The canonical body shipped in livespec-dev-tooling v0.18.0 (its `primary-checkout-commit-refuse-hook-installed` Verifier now accepts both the structural and the legacy body during the fleet migration); core M2-2 brings core's own machinery onto it (core's git-hook-wrapper.sh swapped to the structural body; a shared `just install-commit-refuse-hooks` Installer recipe bootstrap delegates to; the `livespec.primaryPath` write retired). Maintainer-confirmed design (2026-06-25): Option A — one uniform commit-refuse wrapper installed everywhere plus an explicit `livespec.sandboxExempt` marker — chosen over the verifier-sandbox-aware path because the same wrapper carries the lefthook delegation the Fabro sandbox relies on for in-sandbox Red-Green-Replay. The sandbox arming (the orchestrator's Fabro prepare sets `livespec.sandboxExempt true`) landed in livespec-orchestrator-beads-fabro#169. This change is the spec-side half of M2-2, co-landing with the core justfile/wrapper edits in the same PR; it touches existing sections only (no `##` heading added/changed/removed), so tests/heading-coverage.json needs no change.

### Proposed Changes

In `SPECIFICATION/non-functional-requirements.md`:

- §"Workflow discipline — spec-side changes", the **Primary-checkout commit-refuse hook** rule: replace the hook-body paragraph that described `git rev-parse --show-toplevel` compared to `livespec.primaryPath` with a paragraph describing STRUCTURAL detection — the body compares `git rev-parse --git-dir` to `git rev-parse --git-common-dir`, exits 1 when equal (a real primary checkout) and delegates otherwise; it is ARMED ON INSTALL (no `livespec.primaryPath` or other arming step), so there is no fail-open window; it supersedes the earlier `livespec.primaryPath` + `show-toplevel` mechanism, which failed OPEN whenever its arming step was missed (the console-incident root cause: an installed-but-unarmed hook read as protected but silently no-op'd); secondary worktrees differ because a worktree's git-dir is `.git/worktrees/<name>` while its git-common-dir is the primary's `.git`. Add a following paragraph: the one legitimate variation point is an EXPLICIT, DECLARED marker the hook body reads — a Fabro sandbox is a fresh full clone (structurally a primary) that legitimately commits during Red-Green-Replay, so its prepare step sets `git config livespec.sandboxExempt true`; when `true` the refuse branch is skipped (the lefthook delegation still fires); this is the Exemption slot of the Conformance Pattern's Worktree-discipline concern (§"Conformance Pattern"), and outside such a declared sandbox the default is fail-closed.
- §"Commit-refuse hook bootstrap procedure": in the opening sentence, drop "and sets `livespec.primaryPath` in the primary's `.git/config`..." and add that the canonical body is armed on install (it detects the primary structurally) so the bootstrap installs the body and is done, with no arming step. In contract clause 2, drop "AND `livespec.primaryPath` being set in `.git/config`...". In clause 3, drop "AND the correct `livespec.primaryPath`". Replace the illustrative canonical-body `sh` snippet with the structural body: read `git rev-parse --git-dir`, `git rev-parse --git-common-dir`, and `git config --get livespec.sandboxExempt`, refuse (exit 1) when git-dir==git-common-dir and not exempt, else `exec mise exec -- lefthook run --no-auto-install "$(basename "$0")" "$@"`. Replace the trailing paragraph (which described `livespec.primaryPath` storage + `show-toplevel`) with one stating the hook is armed on install (no `livespec.primaryPath` to set); worktrees differ because their git-dir differs from git-common-dir; and a fresh full clone that legitimately commits (a Fabro sandbox) declares `livespec.sandboxExempt true`.
- §"Conformance Pattern", concern #1 (Worktree-discipline): reword the Mechanism / Verifier / Exemption sentences to the confirmed Option-A design — Mechanism: one uniform commit-refuse body installed everywhere (including the sandbox) that refuses structurally (git-dir == git-common-dir) UNLESS the Exemption marker is set, then delegates to lefthook, armed on install with no fail-open window. Verifier: the shared `primary-checkout-commit-refuse-hook-installed` check, which stays simple — it recognizes the canonical body (no sandbox-awareness needed, the marker lives in the body which is present in the sandbox too) and accepts both the structural and the legacy body through the fleet migration. Exemption: a `livespec.sandboxExempt` git-config marker the hook BODY reads (set by the Fabro prepare step). The Contract and Installer sentences are unchanged.

In `SPECIFICATION/contracts.md`:

- §"`master-direct-uncommitted-spec-edits`": in the parenthetical explaining why the sibling commit-refuse invariant cannot prevent the secondary-worktree-on-master bypass, replace "the commit-refuse hook fires only when `git rev-parse --show-toplevel` equals the primary path; secondary worktrees on master pass that comparison and proceed to commit" with "the commit-refuse hook refuses only at the primary checkout — where `git rev-parse --git-dir` equals `git rev-parse --git-common-dir`; a secondary worktree's git-dir differs, so the refuse branch is skipped and the commit proceeds". The invariant's behavior and conclusion (the two checks close the enforcement loop) are unchanged. The §"`primary-checkout-commit-refuse-hook-installed`" invariant itself needs no change — its narration is already body-agnostic (it recognizes the canonical body via a stable marker comment, which both mechanisms carry).
