## Proposal: Conformance Pattern

### Target specification files

- SPECIFICATION/non-functional-requirements.md

### Summary

Add a NEW NON-normative top-level '### Conformance Pattern' section to non-functional-requirements.md, placed immediately after '### Control-Plane console guidance' and before '### Codex dogfooding compatibility'. It records the repeatable cross-repo recipe by which the livespec fleet and its adopters keep a cross-cutting operational policy consistent and provable: the five-slot anatomy of a concern (Contract / Mechanism / Installer / Verifier / Exemption), the reuse-by-default-templatize-only-divergence delivery rule, the consolidated 'just' keystone mandate with its functional/non-functional boundary, the profile boundary plus the declarative '.livespec-fleet-manifest.jsonc' 'adopters' array and 'posture' field, the four-tier enforcement-in-depth, the explicit-exemptions/default-fail-closed hard rule, and the concern registry seeded with concern #1 worktree-discipline and concern #2 cross-harness plugin-resolution. It NAMES and GENERALIZES machinery livespec already runs (the fleet manifest + fleet-conformance check, the copier channel, livespec-dev-tooling shared checks, the pin-and-bump fan-out) rather than duplicating it; like the Dispatcher / grooming / Planning Lane / console guidance it adds no core skill, CLI, or doctor invariant on core's functional surface.

### Motivation

Increment 5, milestone M1 of epic livespec-zs22 (the Conformance Pattern; sub-epic livespec-zs22.7, milestone livespec-zs22.7.2). Source design: research/factory-conformance/cross-repo-conformance-pattern.md (the five slots, profiles, the adopters manifest, four-tier enforcement, the milestones) and research/planning-workflow-gap/planning-lane-design.md (locked decisions: baseline profile not 'factory'; just mandated non-functionally only; fleet pins track latest RELEASE). Lands the PATTERN only; the machinery (M2), fleet dogfood (M3), adopter dogfood (M4), second concern (M5), and four-tier wiring (M6) are tracked as ledger milestones livespec-zs22.7.3 through livespec-zs22.7.7 and are NOT duplicated as spec_commitments here. Placed per the §Boundary litmus as non-functional fleet self-application infrastructure, not a contract a governed consumer inherits.

### Proposed Changes

In SPECIFICATION/non-functional-requirements.md, insert a new top-level '### Conformance Pattern' section immediately after the '### Control-Plane console guidance' section (its '**The just boundary.**' paragraph) and immediately before '### Codex dogfooding compatibility'. It is a sibling '###' under '## Spec' (a cross-cutting fleet methodology gets its own top-level section, peer to the Orchestrator-plugin-ecosystem / Control-Plane-console guidance), not an H4 nested under any of them. No new H2 heading is introduced (the new section is an H3), so tests/heading-coverage.json (which tracks H2 headings only) needs no change. The section opens by framing the Conformance Pattern as fleet self-application infrastructure that NAMES and GENERALIZES existing machinery (§Fleet membership contract, §Shared content sync — copier template, §Shared code sync — livespec-dev-tooling, §Cross-repo coordination — pin-and-bump, and the orchestrator's per-dispatch just check gate) into one fill-in-the-blanks shape, with the same NON-normative-on-core's-functional-surface boundary the Dispatcher / grooming / Planning Lane / console blocks carry (no new core skill / CLI / doctor invariant; realization stays in its owners' specs per §Sibling spec ownership). It then carries bold-led paragraphs: (1) Ubiquitous language — governed repo / livespec fleet / adopter, and the 'baseline' profile named baseline-not-factory because it is a conformance floor; (2) The five-slot anatomy — Contract / Mechanism / Installer (a just recipe) / Verifier (fail-closed, in just check) / Exemption (explicit, declared), a concern unadopted until all five are filled; (3) Reuse by default, templatize only divergence — executable/byte-identical content is reused (pin+import from livespec-dev-tooling or a shared just module), only customizable static scaffold is templatized via copier, applying the §Shared content provenance two-channel partition per concern; (4) The just keystone — just is the single task runner for every governed repo, the always-just-<recipe> entry point lets the same recipe + verifier run everywhere and lets a check assert hooks/CI contain only just, consolidating the piecemeal just references (§Toolchain pins, §Enforcement-suite invocation, §Developer-tooling layout) into one mandate, mandated non-functionally only and never in core's functional spec / the /livespec:* skills / the core↔orchestrator CLI contract; (5) Profiles and the declarative manifest — profile (baseline + additive layers fleet-infra / orchestrator-plugin / app) layered over the manifest's existing repo class, declarative membership via .livespec-fleet-manifest.jsonc gaining an 'adopters' array beside 'fleet' with per-entry 'profile' and per-adopter 'posture' (released / pinned / none); (6) Enforcement in depth (four tiers) — author-time (copier) / commit-time (lefthook → just check) / dispatch-time (orchestrator runs Installer+Verifier before any tenant) / fleet-time (the existing §Fleet membership contract conformance sweep + drift CI), one Verifier definition shared across tiers; (7) The hard rule — every exemption explicit and declared, default fail-closed, a variation point is a marker the Verifier reads not an incidental side effect (the implicit-exemption / fail-open commit-refuse incident is the canonical failure), so the preferred Mechanism is structural / armed-on-install; (8) The concern registry — concern #1 Worktree-discipline and concern #2 cross-harness Plugin-resolution specified in full five-slot form, plus the recognized further members No-shadow-ledger / Terminology-guard / Ledger-closure / Pin-freshness / Archive-on-epic-close named with their one-line contract, each landing through the five slots with its own work. Uses BCP14 keywords consistent with the surrounding document; references existing section titles by §-name only (no cross-spec section citations beyond the established general pointer to livespec-dev-tooling's contracts.md).
