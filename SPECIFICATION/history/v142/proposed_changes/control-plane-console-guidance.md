## Proposal: Control-Plane console guidance

### Target specification files

- SPECIFICATION/non-functional-requirements.md

### Summary

Add a NEW NON-normative top-level '### Control-Plane console guidance' section to non-functional-requirements.md, placed immediately after '#### Planning Lane guidance' (the last child of '### Orchestrator plugin ecosystem') and before '### Codex dogfooding compatibility'. It is a PEER '###' section to '### Orchestrator plugin ecosystem', NOT a '####' nested under it, because the console is the Control Plane — a plane distinct from the Orchestrator Plane — and nesting its guidance under the orchestrator section would contradict the plane separation this track codifies. The section records, for console authors, the repo-agnostic Control-Plane PATTERN: what the console reads, composes, coordinates, and never owns, plus that it is not a required dependency and not a Driver, and the just boundary. Framed identically to the existing Dispatcher / grooming / Planning Lane guidance blocks (core records the pattern; the reference console's own spec owns the realization; no new core skill / CLI / doctor invariant).

### Motivation

Increment 4 of epic livespec-zs22 (research/planning-workflow-gap/planning-lane-design.md §'Increment sequence', item 4: 'Console control-plane contract in both livespec and livespec-console-beads-fabro specs'). Increment 1 framed the three planes and added 'The Control-Plane role' to spec.md §'Workflow planes and the Planning Lane'; this lands the contributor-facing Control-Plane discipline where it belongs per the §Boundary litmus — non-functional fleet infrastructure, not a contract a governed consumer inherits. Honors locked decision 4 (the console is the Control Plane / runner, distinct from a Driver; not yet a required dependency) and locked decision 3 (just mandated non-functionally only, never in core's functional surface). The companion console-side change (the console spec realizing this contract + an extended diagram) lands in livespec-console-beads-fabro as the other half of this cross-repo increment.

### Proposed Changes

In SPECIFICATION/non-functional-requirements.md, insert a new top-level '### Control-Plane console guidance' section immediately after the '#### Planning Lane guidance' block and immediately before '### Codex dogfooding compatibility'. It is a sibling '###' of '### Orchestrator plugin ecosystem' (a separate plane gets a separate top-level section), not an H4 nested inside it. The section opens with the same NON-normative framing as the Dispatcher / grooming / Planning Lane blocks ('core neither names nor verifies any of it'; 'what core records here is the repo-agnostic PATTERN, not the realization'; 'NO new core skill, NO new core CLI, and NO new core doctor invariant'; realization belongs to the reference console's spec, naming livespec-console-beads-fabro's SPECIFICATION/ as the fleet dogfood default), references spec.md §'Workflow planes and the Planning Lane' (its Terminology guard and 'The Control-Plane role') for the architectural frame, and folds the console-is-not-a-Driver distinction into the intro. It then carries six bold-led paragraphs: (1) What the console reads — it OBSERVES read-only the state each other plane already owns (Spec-Plane lifecycle signals from next/doctor/proposed-changes/revise; Orchestrator-Plane ledger + producer state including ready work, in-flight dispatch, blocked-on-human runs, the Dispatcher journal; host/repository hygiene), reaching each plane only through that plane's OWN published surface (spec-side CLIs, the orchestrator list-work-items/next query CLIs, host tools), never private storage; (2) What the console composes — it DERIVES the cross-plane operator views no single plane can produce alone (attention inbox, work cards, timelines, repository-health projections) as read models over observed facts, the join being the console's own product; (3) What the console coordinates — it is the single human interface that INVOKES every plane's operations (spec lifecycle, orchestrator skills, gap/drift consent, factory dispatch) and routes attention, always through the owning plane's published surface (the same one-directional seam discipline the Planning Lane's two seams codify), so the owning plane stays source of truth and actor of record; (4) What the console never owns — NOT /livespec:* spec-mutation semantics, NOT ledger work-item storage, NOT the Dispatcher's factory-execution behavior or a loop's run internals, NOT pull-request merge policy; its authority is confined to its own canonical events, commands, and projections; (5) Not a required dependency — the console MUST NOT become a dependency of the workflow it observes; the spec lifecycle and orchestrator skills stay independently drivable without it; the Control Plane ENRICHES rather than gates, mirroring the standalone-installability rule the orchestrator ecosystem already carries (no plane depends on the console); (6) The just boundary — the console's realization MAY mandate just as its task runner, but just MUST NEVER appear in core's functional spec, the /livespec:* plugin skills, or the core-orchestrator CLI contract. No new H2 heading is introduced (the new section is an H3, and a sixth H4 set lives under it), so tests/heading-coverage.json (which tracks H2 headings only) needs no change.
