# Pin-Bump Reddening Inventory — livespec fleet

## BOTTOM LINE

1. **12 distinct gates** can redden as a downstream consequence of an upstream pin bump, across 8 fleet repos. Adopters (`openbrain`, `resume`, `homelab`) carry **zero** — all three clones are present, and none runs a livespec gate in CI.
2. **Split: 8 are (a) DERIVABLE** — the fan-out could compute and write the fix from static text alone. **4 are (b) NOT** — they need a live orchestrator, a published container image, a reconciled workflow contract, or human code edits.
3. The 8 derivable ones are dominated by **one mechanism**: a new `livespec_dev_tooling/checks/<name>.py` module upstream instantly invalidates the consumer's `justfile` targets array, its `check-<slug>:` recipe body, and its CI matrix — three gates × 7 repos.
4. The fan-out **already solves this class once**: `vendor_jsonc` records trigger `just vendor-update <lib>` and `pyproject_toml_uv_sources` records trigger a `uv.lock` refresh, both landing atomically in the bump commit (`livespec-dev-tooling:.github/actions/bump-pin-rewrite/action.yml:203-227`). That is direct precedent for a "regenerate derived artifacts" step.
5. **Correction to the brief:** I could **not** reproduce the `livespec-console-beads-fabro-7wy` claim. See "Could not verify" — it is load-bearing for how you scope the stricter-upstream work.

### Why master reddens despite auto-merge

Bump PRs are opened with `--auto --rebase` (`livespec-dev-tooling:.github/actions/bump-pin-rewrite/action.yml:24-33`), so in principle a red gate blocks the merge and damage is PR-only. **Empirically that does not hold.** On `livespec` master:

```
f5380aa8  chore(deps): bump livespec-dev-tooling pin to v0.31.3            <- merged
5a8f0c0c  ...
243bfde8  ...
38714bf2  ...
d8f18376  chore(templates): restamp canonical-slugs.yml for ... v0.31.3    <- repair, 4 commits later
```

`check-canonical-slugs-projection` was already wired into both `just check` (2 hits) and `ci.yml` (1 hit) at `d8f18376~1`, yet `f5380aa8` landed anyway. **I could not determine the mechanism** — Actions run history for those SHAs has aged out (`gh run list --commit <sha>` returns empty for all five). Candidates I could not distinguish: a warm uv cache resolving the old dev-tooling during the PR run, or a required-checks configuration not including the failing context. **This is the most decision-relevant open question here** — if auto-merge genuinely gates, this class is PR-only annoyance; if not, it reddens master, trips `check-master-ci-green`, and blocks the repair push.

## LATENT — stricter-upstream-check hazards (quiet today, will bite on a future bump)

- **`check-ci-matrix-completeness` warn-lever asymmetry.** Repos without `LIVESPEC_FAIL_IF_CI_MATRIX_GAPS_EXIST` accumulate CI-coverage gaps silently and exit 0. Flipping the lever in a wiring PR fires every accumulated gap at once. `livespec-dev-tooling:livespec_dev_tooling/checks/ci_matrix_completeness.py:39-48`.

- **`check-behavior-scenario-link` severity flip.** `LIVESPEC_BEHAVIOR_SCENARIO_LINK` defaults to **`warn`** in core (`livespec:justfile:675-681`) but to **`fail`** in the console's Rust port (`livespec-console-beads-fabro:crates/console-spec-check/src/lib.rs:41-42`). Core's comment says it "flips to fail later"; when it does, every core clause lacking a `tests/heading-coverage.json` link fires at once.

- **`file_lloc` hard-gate lever.** `livespec` sets `file_lloc_hard_gate = true` (`livespec:pyproject.toml`), extending the >250-LLOC hard fail to its whole first-party universe. Un-armed repos get warnings only. A future dev-tooling release making the flip the default reddens every un-burned-down consumer.

- **`copier-update-drift` workflow — dormant, will arm.** Present in `livespec-orchestrator-beads-fabro` and `livespec-orchestrator-git-jsonl` but reduced to `workflow_dispatch` only, with an in-file note to restore the `pull_request + push + schedule` block once a real release tag exists (`livespec-orchestrator-beads-fabro:.github/workflows/copier-update-drift.yml:24-33`). That precondition is now met — core is at v0.18.0. Restoring triggers turns **every core template change** into a drift failure across both orchestrators, refreshable only by a `copier update --vcs-ref=master` 3-way merge (**(b) — human judgment on conflicts**). `livespec-orchestrator-git-jsonl:.copier-answers.yml:9` still reads `_commit: v0.4.0`.

- **`console-spec-check` gap-id golden — silent drift, the INVERSE hazard.** `crates/console-spec-check/tests/data/parity_golden.json` is a frozen 8-vector golden asserting the Rust gap-id port matches core's `spec_clauses.py`. But the test compares Rust-against-golden, **never against live Python** (`livespec-console-beads-fabro:crates/console-spec-check/src/tests.rs:18-31`). If core changes `derive_gap_id` or the clause-extraction regex, this test stays **green** while the two implementations silently diverge, producing mismatched gap-ids across the family. A correctness hazard a pin bump will *not* surface — worth a work-item independent of the reddening question.

- **Worktree-discipline pack byte-identity — dormant.** Verified: **no fleet repo tracks** `dev-tooling/worktree-lib.sh`, `branch-protection.sh`, `worktree.just`, or `branch-protection.just`. The arm is skipped everywhere. The first repo to install the pack arms byte-identity against four more dev-tooling constants (**(a) DERIVABLE** — `install_worktree_pack` writes them).
