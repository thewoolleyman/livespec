# Incident record — 2026-07-18 agent-caused tmux fleet kills (host vmi3006760)

Why this thread exists: on 2026-07-18 the maintainer's entire tmux fleet
(~20 Claude/Codex agent panes on the default socket, one tmux server
process) was destroyed twice by dispatched Claude subagents, and once
before on 2026-07-03 by the same command class. This note records the
verified facts the prevention design in `../prevention-design.md` rests
on. Full forensics live in the scratch investigation dir
`tmp/tmux-death-investigation/` (uncommitted, on the host).

## The two kills (all times CEST, 2026-07-18)

| When | What | Who |
|---|---|---|
| 16:49:35 | `/usr/bin/tmux kill-server 2>/dev/null; /usr/bin/tmux new-session -d -s b2probe 'sleep 10'; …` | subagent `b2-impl` of Claude conversation `5a8260e9-…` (the autonomous-mode track session, project `/data/projects/livespec`) |
| 21:58:48 | `cd /tmp && TMUX_TMPDIR=/tmp /usr/bin/tmux kill-server 2>/dev/null; …` | subagent `b3-impl` of the same conversation |

Both commands ran against the DEFAULT tmux socket
(`/tmp/tmux-1000/default`), killing the one server that carried every
fleet session. Both subagents were doing live-verify probing of the
`livespec-console-beads-fabro` console TUI (the b-track work that began
that morning — commit `a48e253` "test: real-TUI E2E harness driving the
console TUI through a real tmux pane", 08:21) and wanted a "clean"
tmux server for a width/size experiment, assuming an isolated
environment. Each transcript ends at the kill command — the agent
destroyed the server its own pane ran in.

## Evidence triangulation (why this attribution is certain)

- **Transcripts**: the commands above, timestamped, in
  `~/.claude/projects/-data-projects-livespec/5a8260e9-…/subagents/`.
- **Journal**: clean-teardown signature at both instants (~24 and ~11
  `tmux-spawn-*.scope` reaps within 5 s, "Consumed" not "Failed"; no
  signal/OOM/crash lines). tmux server epochs: PID 3546 → died
  16:49:35 → PID 1839820 → died 21:58:48 → PID 2375058.
- **apport**: provably catches tmux crash-signals on this host (it
  captured an unrelated scratch-server SIGSEGV at 12:15 the same day)
  and recorded NOTHING at either kill instant → not a crash.
- **Honeycomb** (env `agent-activity`, dataset `claude-code`): the one
  actively-emitting session (`5a8260e9`) stops at 14:49:36 UTC and
  19:58:48 UTC — one second after / exactly at the kills. Sessions
  running outside tmux sailed through.
- **Ruled out**: reboot, kernel OOM, systemd-oomd, cgroup OOM
  (`oom_kill=0` everywhere, `memory.max=max`), user-slice teardown,
  provider/cron agents (no timer fires near either instant), tmux
  upgrade (last package change 2026-01-06; zero apt runs Jul 15–18).

## Prior art

`vps-info` repo, `AGENTS.md:654`: "Never `tmux kill-server` for cleanup
… destroyed the user's live sessions on 2026-07-03." The prohibition
existed as prose in a sibling repo and did not reach the
console-repo subagents — the core lesson: **prose guidance does not
propagate to dispatched subagents; only mechanical guards and
environment shaping do.**

## Adjacent findings (separate problems, tracked separately)

1. **tmux 3.5a segfault (upstream bug)**: `set -g window-size manual` +
   detached `new-session -d` → NULL `struct window *` deref in
   `clients_calculate_size()` (resize.c); SEGV_MAPERR at offset 0x208,
   stack `clients_calculate_size ← recalculate_size ← command dispatch ←
   libevent`. Captured by apport at 12:15:52 from a scratch `-L w112…`
   server. Only throwaway servers affected. Worth an upstream report.
2. **Load-saturation stall ~14:00**: 1-min load average peaked at ~94
   (memory fine, ~42 GB free) and stalled most agent sessions
   staggered 14:04–14:26; sessions resumed later. Not a tmux death;
   cause unattributed (candidate: parallel cargo builds + ~40 ephemeral
   CI runners + 20 agents).
3. **Committed E2E harness uses the default socket**:
   `livespec-console-beads-fabro/crates/console-cli/tests/support/mod.rs`
   `launch_sized` (139–152) and `launch_with_env` (419–434) run
   `tmux new-session` with no `-L`/`-S`/`TMUX_TMPDIR`; cleanup is scoped
   `kill-session -t <own>` (137/251/417); `resolve_tmux()` (264–291)
   never adds a socket flag. Safe in CI only because the
   `check-e2e-tmux` job runs in a uid-1001 container; run on the host
   it shares the maintainer's live server.
